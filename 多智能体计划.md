# 多智能体计划

## 目标

在现有 `ChatAgent + runtime blocks + hooks + orchestration queue core` 基础上，落地一套**主智能体 / 子智能体同构**的多智能体体系。

这套体系不是隐藏 runtime 注入，而是：

- 主智能体可以创建、查看、调度、删除子智能体
- 子智能体可以独立运行、独立对话、独立上报
- 调度器负责公共消息池、消息排队、原子步骤后插入、空闲唤醒
- 前端可见主会话和各个子会话之间的关系

## 当前已落地状态

- 已落地子智能体 definition registry：
  - `backend/src/services/subagents/SubagentDefinitionRegistry.js`
- 已落地子智能体 runtime factory：
  - `backend/src/services/subagents/AgentRuntimeFactory.js`
- 已落地上层 supervisor：
  - `backend/src/services/orchestration/OrchestratorSupervisorService.js`
- 已落地空闲唤醒与后台运行：
  - `backend/src/services/orchestration/AgentWakeDispatcher.js`
  - `backend/src/services/orchestration/ConversationAgentRuntimeService.js`
- 已落地持久化 orchestration store：
  - `backend/src/services/orchestration/SqliteOrchestratorStore.js`
- 前台主对话与子对话都已按 `conversationId` 选择对应 runtime：
  - 主对话走主 `ChatAgent`
  - 子对话走各自 scoped `ChatAgent`
- 调度器 queue / pool / agent 状态已入库
- 前端已支持：
  - 历史列表区分主对话 / fork / subagent
  - 子对话禁止 fork
  - 流式中切换查看别的对话，后台继续写回原 `conversationId` 本地缓存
  - 子对话 developer prompt 前端只读

## 已确认需求

### 1. 主智能体和子智能体必须同构

- 子智能体不是一个特殊轻量对象，而是和主智能体一样的 agent runtime
- 实现上应尽量复用同一个：
  - `ChatAgent`
  - tool loop
  - approval 机制
  - runtime block 注入
  - hooks runtime
  - 自动记忆 recall
- 区别主要体现在：
  - agent definition
  - 专属 tools / skills / prompt
  - 权限限制

### 2. 子智能体定义位置

- 子智能体定义统一放在 `backend/src/subagents/`
- 每个子智能体一个独立目录，例如：
  - `backend/src/subagents/researcher/`
  - `backend/src/subagents/planner/`
  - `backend/src/subagents/coder/`
- 每个目录至少包含：
  - `prompt.*`
  - `tools/`
  - `skills/`
  - `hooks/`
  - `definition.*`
- 不需要：
  - `AGENTS.md`
  - `soul.md`
  - 用户手写自由提示词注入层

这里的 `skills/` 目录不是本地 skill 正文注入目录，而是子智能体自己的 skill 选择说明位。

真实 skill 来源统一仍然是：

- 全局 `.yyz/skills/`
- 工作区 `.yyz/skills/`

也就是说子智能体不会去读取自己目录下的 `skills/*.md` 全文塞进 system。

子智能体真正使用的是当前 session 的全局选择性 skills：

- 主对话和子对话共享同一套已选 skills
- 子智能体 definition 不再声明 skills 选择
- system 只注入 skill 的 name / description / index
- 需要完整 skill 内容时主动调用 `skill_view`

### 3. 子智能体的通用能力与限制

- 子智能体保留和主智能体相同的通用运行链路：
  - workplace prompt
  - 通用运行规范
  - runtime block 体系
  - 文件解析注入
  - 记忆 recall 注入
- 子智能体注入自己的类型 `prompt.md`
- 子智能体不注入工作区 `SOUL.md / AGENTS.md`
- tools / hooks 改成“shared base 继承 + 子目录增量”
  - definition 只正向声明 `inheritedBaseToolNames / inheritedBaseHookNames`
  - 子目录仍可独立追加 `tools/` 和 `hooks/`
  - 具体 `hooks/definitions` 由每个子智能体目录自行提供
- 子智能体**不允许修改长期记忆**
  - 可以吃 recall
  - 不允许暴露 memory write 工具
  - 不允许借其他工具绕过写记忆

### 4. 调度消息和体系消息的表现形式

- 所有调度体系内的消息，插入到会话时一律表现为：
  - `role = "user"`
- 但通过 `meta.kind` 和 `meta.subtype` 区分来源和类型
- 前端可以基于 `meta.kind` 做专门渲染
- 这类消息不是 runtime block，不是隐藏消息，是**真正可见的消息**

建议的消息分类：

- 普通用户输入
  - `role=user`
  - `meta.kind` 为空，或保留 `user_input`
- 主子智能体调度消息
  - `role=user`
  - `meta.kind=orchestrator_message`
- 主智能体派发任务给某个子智能体
  - `meta.kind=orchestrator_message`
  - `meta.subtype=agent_dispatch`
- 子智能体阶段上报 / 完成上报
  - `meta.kind=orchestrator_message`
  - `meta.subtype=agent_report`
- 公共池轻量广播通知
  - `meta.kind=orchestrator_message`
  - `meta.subtype=pool_notice`
- 唤醒消息
  - `meta.kind=orchestrator_message`
  - `meta.subtype=agent_wakeup`

### 5. 公共消息池不是“广播临时文本”，而是真实运行时存储

- 公共消息池必须真实存储消息
- 公共消息池不能只做内存态；必须支持服务重启后的恢复
- 正式实现采用：
  - 数据库作为真实来源
  - 内存作为活跃会话的热缓存
- 主智能体和子智能体都能查看公共池
- 来自公共消息池的自动投递消息必须明确分两层：
  - 轻量层：`name/title`
  - 全量层：详细正文 / detail / payload
- 子智能体默认不直接收到全量正文，只收到轻量 `name/title`
- 如果子智能体要看详情，需要主动调用查看公共池的工具
- 主智能体收到的是全量详细消息
- 前端上也要允许把这两类消息渲染成不同密度

公共池本体需要包含：

- entry id
- sequence
- sourceAgentId
- subtype
- title
- summaryLines
- detailLines
- payload
- metadata
- createdAt

公共池持久化原则：

- 公共池 entry 采用 append-only
- 广播消息和 queue 插入只是公共池 entry 的派生动作
- 服务重启后要能恢复：
  - 公共池最新内容
  - delivery / queue 相关状态
  - 活跃会话热缓存

### 6. 现有调度器只是底层 queue core，还需要上层调度器

当前已有：

- `OrchestratorSchedulerService`
  - session
  - agent registration
  - publicPool
  - queue
  - atomic step
  - flush ready insertions

仍需补充一个上层 orchestrator supervisor，负责：

- agent 路由
- 广播策略
- 公共池 entry -> 广播消息的转换
- 主智能体创建子智能体
- 结束信号收口
- 唤醒 idle agent

### 7. 唤醒机制必须补上

调度器不能只会“目标 agent 忙时排队”。

还必须支持：

- 目标 agent 忙
  - queue，等原子步骤结束后插入
- 目标 agent 空闲
  - 直接投递并触发 agent 开始运行

主要用途：

- 子智能体上报时，主智能体可能当前不在工作
- 调度器需要能把消息投递给空闲主智能体并唤醒它
- 主智能体派发任务时，目标子智能体也可能当前不在工作
- 调度器需要能把消息投递给空闲子智能体并唤醒它
- 也就是说，唤醒机制必须同时覆盖：
  - 主智能体
  - 子智能体

### 8. 原子步骤约束仍然保留

- 任何调度插入都不能打断当前原子步骤
- 所有“汇报给主智能体”的动作必须在原子步骤结束后插入
- 现有 `beginAtomicStep / finishAtomicStep / queue / flush` 仍然是底层准则
- 主智能体调度子智能体也必须走同一套 queue insert 机制
- 不允许主智能体绕过 scheduler 直接把任务塞进子智能体会话
- 也就是说：
  - 主 -> 子 dispatch：走 queue / idle wake
  - 子 -> 主 report：走 queue / idle wake
  - 公共池 notice：走 queue / idle wake

### 9. 主智能体工具能力

主智能体至少要有这些工具：

- 创建子智能体
- 查询当前对话下的子智能体列表
- 查看子智能体状态
- 给指定子智能体发送调度消息
- 查看公共消息池
- 删除指定子智能体

其中删除行为要求：

- 删除子智能体时，除公共池 entry 外，其余与该子智能体相关的内容全部删除
- 删除当前对话下的主子关系绑定
- 删除该子智能体聊天记录 / 会话 / 运行状态
- 不删除该子智能体写入公共池的历史消息
- 删除后的结果是：
  - 不再作为活跃子智能体继续调度
  - 用户和主智能体都不能再进入该子会话
  - 但公共池里的项目流程记录继续保留

### 10. 用户能力

- 用户也可以直接和子智能体对话
- 用户也可以删除子智能体
- 用户可以从主对话进入子对话
- 不需要在主对话里额外提供“直接向某个子智能体发调度消息”的用户入口
- 用户可见体系消息，但前端需要基于 `meta.kind` 做区分渲染，避免误解

### 11. 前端要求

前端需要支持：

- 主对话展示并引出当前对话下的所有子对话
- 主智能体页面可看到：
  - 子智能体列表
  - 类型
  - 状态
  - 最近活跃时间
  - 是否忙碌
- 可从主对话跳转进子对话
- 可从子对话返回主对话
- 用户与子智能体交互时，直接进入该子对话正常聊天，不需要主对话里的直发调度输入框
- 主 / 子对话都能显示调度消息
- 基于 `meta.kind/meta.subtype` 做不同样式渲染
- 流式生成期间允许切换当前查看的对话，后台运行不断开
- 这项主要是前端状态管理改造：
  - 流式事件和增量消息必须按 `conversationId` 归属
  - 后端接口原则上可复用现有流式链路，不需要先改协议
- 删除子智能体后，对应子对话入口消失

### 12. 数量限制

- 每个对话最多 5 个子智能体

## 架构理解

## A. Runtime 层

主智能体和子智能体都应该跑在同一个 runtime 模型上：

- `ChatAgent`
- runtime blocks
- hooks runtime
- same tool loop
- same approval flow

差异由 `agent definition` 注入，而不是再复制一套 Agent 类。

## B. Definition 层

每个子智能体目录负责声明：

- agentType
- displayName
- system prompt
- tools loader
- skills loader
- hooks loader
- capability flags

其中 `skills loader` 的真实语义是：

- 读取全局 / 工作区 skill catalog
- 基于当前会话已选 skills 生成 skills index prompt
- 不把 skill 正文直接注入 system
- 完整正文由 `skill_view` 按需查看

建议增加：

- `SubagentDefinitionRegistry`
  - 负责扫描 `backend/src/subagents/*`
  - 注册子智能体类型
  - 加载子智能体自己的 `prompt.md`
  - 构建“base 继承 + 本地增量”的 tool registry / hook registry

## C. 调度层

调度层拆为两层：

- `OrchestratorSchedulerService`
  - 底层 queue core
  - 公共池
  - atomic step
  - flush
- `OrchestratorSupervisorService`
  - 上层多智能体编排
  - 创建 agent
  - 路由消息
  - 管理广播
  - 主子关系
  - 结束信号
  - 唤醒机制

## D. 公共消息池层

建议新增一个更明确的运行时概念：

- `SharedMessagePoolRuntime`

职责：

- 维护对话级公共池
- 提供 append/read/list API
- 供主智能体和子智能体工具访问
- 被调度器用于生成轻量广播和全量广播
- 轻量广播消息体默认只带 `name/title`
- 全量广播消息体带 `detailLines/payload`
- 持久化公共池 entry 和 delivery 状态
- 启动时从数据库恢复活跃会话热缓存
- 保证子智能体删除后历史公共池 entry 仍可查看

## E. 唤醒层

建议新增：

- `AgentWakeDispatcher`

职责：

- 判断目标 agent 当前是否有活跃 run
- 空闲时触发一次新的 run
- 忙碌时继续走 queue
- 同时适用于：
  - 主智能体被子智能体上报唤醒
  - 子智能体被主智能体 dispatch 唤醒
  - 任意 agent 被公共池 notice 唤醒

## 数据模型建议

### 1. Conversation -> Subagent 关系

建议新增关系存储：

- conversationId
- agentId
- agentType
- displayName
- status
- createdAt
- updatedAt

### 2. 子智能体会话

子智能体聊天记录可以继续复用现有 history 体系，但要补：

- parentConversationId
- source=subagent
- metadata 中标记：
  - `ownerConversationId`
  - `agentId`
  - `agentType`
  - `isSubagent=true`

- 子智能体会话禁止 fork
- 后端需要显式拒绝：
  - 从子智能体会话 fork
  - 把 fork 工具暴露给子智能体

子智能体删除语义：

- 删除时直接删除：
  - 子智能体会话
  - 主子关系
  - 运行时活跃状态
- 不保留子智能体自身聊天记录
- 但公共池 entry 保留 `sourceAgentId` 与历史正文，继续作为项目流程事实可查
- 删除后该子智能体不可恢复原会话，也不允许保留为归档会话

### 2.1 公共消息池持久化模型

建议至少拆两张表：

- `orchestrator_pool_entries`
  - `id`
  - `session_id`
  - `sequence`
  - `source_agent_id`
  - `subtype`
  - `title`
  - `summary_lines_json`
  - `detail_lines_json`
  - `payload_json`
  - `metadata_json`
  - `created_at`
- `orchestrator_pool_deliveries`
  - `id`
  - `pool_entry_id`
  - `target_agent_id`
  - `delivery_kind`
  - `delivery_mode`
  - `status`
  - `queued_message_id`
  - `delivered_at`
  - `consumed_at`

语义约束：

- `orchestrator_pool_entries` 代表项目流程事实
- `orchestrator_pool_deliveries` 代表“这条事实通知过谁、到了哪一步”
- 即使子智能体被删除，历史 `pool_entries` 也必须保留
- 重启恢复时以数据库为真源重建内存热缓存

### 3. 调度消息元数据

建议统一：

- `meta.kind=orchestrator_message`
- `meta.subtype`
- `meta.sourceAgentId`
- `meta.targetAgentId`
- `meta.poolEntryId`
- `meta.deliveryMode`
- `meta.broadcastMode`

## 工具与权限设计

### 主智能体工具

- `subagent_types_list`
- `subagent_create`
- `subagent_list`
- `subagent_inspect`
- `subagent_dispatch`
- `subagent_delete`
- `pool_list`
- `pool_read`

### 子智能体工具

- `pool_list`
- `pool_read`
- `pool_report`
- 自己专属目录里的业务工具
- 不暴露 fork 当前会话能力

### 禁止给子智能体的工具

- 所有长期记忆写工具
- 主智能体专属调度管理工具
- 全局破坏性管理工具

## Hook 设计补充

- Hook runtime 逻辑复用现有：
  - `HookRegistry`
  - `HookBlockBuilder`
  - `hooks.runtime-block`
- 但每个子智能体独立加载自己目录下的 hook 定义
- 主智能体和不同子智能体的 hook 集合互不污染

建议模式：

- `backend/src/subagents/<agentType>/hooks/*.hook.js`
- `SubagentDefinitionRegistry` 在构建 agent runtime 时装配独立 `HookRegistry`

## 记忆策略补充

- 子智能体保留自动 recall 注入
- 子智能体禁用 memory write 工具
- 如果需要更强约束，系统提示词也明确写：
  - 可利用已有长期记忆
  - 禁止修改长期记忆

## 前端展示设计

### 主对话视图

- 顶部或侧栏展示本对话下的子智能体列表
- 每个子智能体展示：
  - 名称
  - 类型
  - 状态
  - 最近活跃时间
- 支持：
  - 点击进入子对话
  - 删除子智能体

### 子对话视图

- 结构和主对话相同，允许用户直接输入
- 会显示调度器插入消息
- 可查看公共池相关消息提醒
- 子对话不提供 fork 入口
- 流式生成期间允许切走查看别的对话，原子智能体 run 在后台继续

### 调度消息渲染

- 所有体系消息都在聊天流里显示
- 但基于 `meta.kind/meta.subtype` 使用专门样式
- 用户真实输入和调度消息必须视觉区分

## 分阶段实施计划

## 阶段 1：Definition 与 Runtime 复用层

- 建 `backend/src/subagents/`
- 实现 `SubagentDefinitionRegistry`
- 约定每个子智能体目录的结构
- 让子智能体复用 `ChatAgent` 主运行逻辑
- 支持独立 tools / skills / hooks / prompt
- 给子智能体加上“可 recall、不可写记忆”的权限约束

## 阶段 2：上层 Orchestrator Supervisor

- 在现有 `OrchestratorSchedulerService` 之上加：
  - 创建子智能体
  - 查询子智能体
  - 路由调度消息
  - 上报主智能体
  - fallback 结束信号
- 明确所有主->子 dispatch 也走 queue insert，不允许旁路
- 保留原子步骤后插入规则

## 阶段 3：唤醒机制

- 增加 idle / busy 判断
- 空闲 agent 时允许 supervisor 直接投递并唤醒
- 忙碌 agent 时继续 queue
- 主智能体和子智能体都适用同一套唤醒机制

## 阶段 4：公共消息池 runtime 与工具

- 把公共池明确成共享 runtime
- 以数据库为真源，内存做热缓存
- 提供：
  - `pool_list`
  - `pool_read`
  - `pool_report`
- 实现：
  - 子智能体收到轻量 `name/title`
  - 主智能体收到全量 detail/payload
  - 需要详情的子智能体自行 `pool_read`
  - 重启后恢复公共池和 delivery 状态

## 阶段 5：主智能体工具与限制

- 补：
  - `subagent_types_list`
  - `subagent_create`
  - `subagent_list`
  - `subagent_dispatch`
  - `subagent_delete`
- 加每对话上限 5 个子智能体

## 阶段 6：前端主子会话联动

- 主对话中展示子智能体入口
- 子对话独立可聊
- 流式期间允许切换查看其他对话，后台 run 不断开
- 调度消息特殊渲染
- 删除子智能体时删除其会话历史，仅保留公共池 entry
- 子智能体不提供 fork 入口

## 阶段 7：持久化与稳定性

- 子智能体关系持久化
- 公共池持久化与重启恢复
- 恢复主对话时同步恢复子智能体列表
- 已删除子智能体不恢复聊天历史，只保留公共池事实
- 加基本端到端验证

## 当前推荐实现顺序

建议按下面顺序做，而不是一上来全铺开：

1. `SubagentDefinitionRegistry`
2. `OrchestratorSupervisorService`
3. `AgentWakeDispatcher`
4. 主智能体调度工具
5. 公共池工具
6. 前端主子对话联动

## 关键风险

- 如果子智能体不走同构 runtime，后面会形成两套 agent 体系，维护成本会爆炸
- 如果公共池只是广播文本而不是可查询运行时，子智能体会被迫吃全量注入，灵活性差
- 如果公共池不持久化，服务重启后多智能体流程会直接断裂
- 如果不补唤醒机制，空闲主智能体收不到子智能体上报
- 如果 hooks 不按 agent 隔离，不同子智能体工具差异会把 hooks 搞乱
- 如果调度消息不统一走 `role=user + meta.kind`，前端渲染和历史筛选会越来越脏
- 如果删除子智能体时把公共池 entry 一起删掉，会丢项目流程证据
- 如果允许子智能体会话 fork，会把主子编排关系和 lineage 搞乱
